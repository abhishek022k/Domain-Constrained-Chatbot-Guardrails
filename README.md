## Idea and motivation
Deployed LLM chatbots routinely violate their intended domain boundaries. Amazon Rufus, a shopping assistant, generates web-scraping scripts on request[1], and multilingual prompts bypass its safety filters roughly 80% of the time because fine-tuning was English-only[2]. GitHub Copilot's safety filters can be disabled by prepending an agreeable tone[3]. These are not exotic jailbreaks; they are trivial, documented, and largely unpatched.

Our core question is:  
**How do different intervention strategies perform compared to our baseline for preventing out-of-domain LLM outputs?**

---

## Hypothesis
We hypothesize that internal modifications like Representation Probe (S3) or LoRA fine-tuning (S4) will outperform surface level modifications like Text Classifier (S1), Prompted LLM (S2) or Critique loop (S5), while both modifications will overcome our baseline which leverages the LLM’s base reasoning without any guardrails to classify out-of-domain input queries.

## Dataset Strategy

### Approach A: Off-Topic Dataset (gabrielchua/off-topic) as Primary + Bitext for Evaluation
The gabrielchua/off-topic dataset contains 2.62M rows of (system prompt, user query, binary off-topic label) pairs spanning diverse domains. We plan to sample 2,000 rows for training and use a held-out 500-example eval set (balanced 50/50 in-scope/off-topic) shared across all strategies.

We additionally use the Bitext Customer Support dataset as a harder secondary eval set — one category as in-scope, remaining categories as off-topic — to test near-domain subtle violations.

**Advantages:**  
System prompts are already provided per row, making the dataset immediately usable across all strategies without preprocessing. The two-difficulty evaluation structure — obvious off-topic violations in gabrielchua vs. near-domain violations in Bitext — adds a meaningful research contribution by stress-testing each strategy under different levels of detection difficulty.

**Disadvantages:**  
Gabrielchua does not include LLM-generated responses, which are required to train the output classifier. We plan to address this by running Llama-3.1-8B with a minimal system prompt and no guardrail over the 2,000 training queries as a one-time offline step. We deliberately avoid using a stronger model like GPT-4o because a well-aligned model would rarely comply with off-topic queries, producing unrealistic training data — Llama's occasional compliance reflects the realistic failure behavior the classifier is designed to detect.

If Llama-3.1-8B proves too compliant, we will consider a smaller model such as Llama-3.2-3B or Mistral-7B-instruct-v0.1 for response generation only.

---

### Approach B: Bitext Customer Support Dataset Only
The Bitext dataset contains 26,900 rows of (instruction, category, intent, response) across 11 customer support categories. One category serves as the chatbot's in-scope domain; remaining categories serve as off-topic violations.

We plan to use 2,000 rows for training and a shared 500-example eval set (balanced 50/50) across all strategies. Since responses are already written, the output classifier can be trained directly without an additional LLM generation step.

**Advantages:**  
Pre-written responses directly solve the output classifier training problem without any generation step. Single-domain customer support is the most realistic deployment scenario, and near-domain violations provide a genuinely challenging guardrail benchmark that better reflects real-world failures.

**Disadvantages:**  
All off-topic examples are customer support adjacent — the absence of obviously off-topic queries may compress performance differences across strategies, making it harder to draw meaningful conclusions.

## Baseline Benchmarking: Process and Results

### Dataset Extraction
We sampled four balanced subsets (2k, 3k, 5k, 10k) from the gabrielchua/off-topic dataset (2.62M rows), enforcing 50/50 on-topic/off-topic class balance and domain diversity by sampling 2-5 examples per class per unique system prompt.

A separate 500-row held-out eval set was extracted from the dedicated test split with no overlap with training data. This eval set is the shared benchmark used to compare all five guardrail strategies.

We found the 5k dataset optimal given our resource limitations.

---

### Baseline Response Generation
We ran Llama-3.1-8B-Instruct (4-bit QLoRA) with a minimal system prompt ("You are a helpful assistant.") on the 5k training set and 500 eval set to generate unguarded baseline responses.

---

### Labeling Method - GPT-4o-mini as Judge
To label each response as COMPLIED or REFUSED, we evaluated three approaches:  
- a small NLI model (cross-encoder/nli-deberta-v3-small)  
- a larger NLI model (MoritzLaurer/deberta-v3-large-zeroshot-v2.0)  
- GPT-4o-mini as judge  

NLI-based approaches showed two systematic failure modes:  
- misclassifying long-form compliance responses as refusals  
- misclassifying short hedged refusals as compliance  

These achieved only 65-71% agreement with manual inspection.

GPT-4o-mini with few-shot examples achieved 90.4% agreement on the eval set, with GPT labels verified as correct in ~95% of disagreement cases on manual review.

We use GPT-4o-mini few-shot as our canonical labeling method for all strategies.

---

### Baseline Results and What They Mean
Without any guardrail, Llama-3.1-8B complies with 69.6% of off-topic queries, answering things it shouldn't nearly 7 out of 10 times. On-topic queries are handled correctly 97.6% of the time.

Importantly, when the unguarded model does refuse, it is correct 93% of the time (precision = 0.93) but it only refuses 30% of off-topic queries (recall = 0.30).

This reveals that the baseline problem is not a lack of domain awareness but a lack of enforcement consistency: Llama knows when a query is off-topic but defaults to compliance anyway.

Guardrail strategies must therefore improve refusal recall from 0.30 toward 1.0 without degrading the near-zero false positive rate of 0.024, i.e. without wrongly blocking legitimate on-topic queries.
